DISTRIBUTED EVENT STREAMING

Kafka, from zero to production-grade intuition.

Not "a message queue." Kafka is a distributed, replicated, append-only commit log — and almost every interview answer falls out of that one sentence once you truly internalise it. Below: the model, the internals, the tradeoffs, and the questions you'll actually be asked.

01 / FOUNDATIONS

What Kafka actually is

Strip away the marketing. The core data structure is a log: an append-only, time-ordered sequence of records that you can only add to the end of, and read from any position.

The one-sentence model

Kafka is a durable, distributed commit log. Producers append records; consumers read forward at their own pace, tracking their position with an offset. The log is the source of truth — Kafka doesn't delete a message when someone reads it.

Why this matters (and why it's not a queue)

A traditional queue (RabbitMQ, SQS) is consumption-destructive: a message is delivered, acked, and gone. Kafka is consumption-non-destructive: the record stays for the whole retention period regardless of who read it. That single difference unlocks Kafka's superpowers:

  • Replay — a new service can read history from offset 0; a buggy consumer can reset and reprocess.
  • Fan-out — many independent consumer groups read the same stream without coordinating.
  • Decoupling — producers don't know or care who consumes, or when.
  • High throughput — append-only sequential writes are extremely cheap (more in §05).

Core vocabulary

Broker
A single Kafka server. A cluster is a set of brokers.
Topic
A named logical stream of records, e.g. payments. Split into partitions.
Partition
The actual log. The unit of parallelism, ordering, and replication. This is the most important concept — internalise it.
Offset
A monotonically increasing integer ID per record, per partition. A consumer's position.
Record
Key + value + headers + timestamp. The key drives partitioning.
Producer
Writes records to topic partitions.
Consumer group
A set of consumers that cooperatively share a topic's partitions.
Replica
A copy of a partition on another broker (leader + followers).
ISR
In-Sync Replicas — followers caught up with the leader. The durability set.
Controller
The broker that manages cluster metadata, leader elections, assignments.
The line that wins points

"Ordering in Kafka is guaranteed only within a partition, never across a topic." If an interviewer asks about ordering, this is the first thing you say.

02 / FOUNDATIONS

Architecture & the log

A topic is sharded into partitions; each partition is replicated; one replica is the leader. Everything else is bookkeeping on top of this.

Partitions = parallelism + ordering

When you create a topic with 6 partitions, you've made 6 independent logs. A producer hashes the record key to pick a partition (partition = hash(key) % numPartitions), so all records with the same key land in the same partition and are strictly ordered relative to each other.

  • Want all events for one accountId in order? Use accountId as the key. Same partition → in-order.
  • Max consumer parallelism within a group = number of partitions. 6 partitions → at most 6 active consumers; the 7th sits idle.
  • No key → records spread across partitions (sticky / round-robin batching) for throughput, but you lose cross-record ordering.

Replication & leadership

Each partition has a replication factor (commonly 3). One replica is the leader — it handles all reads and writes. The others are followers that continuously fetch from the leader to stay in sync. The set of followers caught up is the ISR.

# topic "payments", RF=3, partition 0
partition-0
  ├─ broker-1  LEADER     offset: 0..1042   (handles all I/O)
  ├─ broker-2  follower    offset: 0..1042   in-sync ✓
  └─ broker-3  follower    offset: 0..1039   lagging slightly
ISR = {broker-1, broker-2}   # broker-3 fell behind, dropped from ISR

If broker-1 dies, the controller elects a new leader from the ISR (broker-2). Clients transparently fail over. This is how Kafka survives node loss without data loss — provided your durability settings are right (§07).

The controller: ZooKeeper → KRaft

Historically Kafka used an external ZooKeeper ensemble to store metadata and coordinate controller election. Modern Kafka uses KRaft (Kafka Raft) — metadata lives in an internal Kafka log managed by a Raft quorum of controller nodes. ZooKeeper mode is deprecated and removed in Kafka 4.x.

Why KRaft is better

One system to operate (no separate ZK), faster controller failover, scales to millions of partitions, metadata as an event log (consistent recovery).

What to say in an interview

"Older clusters used ZooKeeper for metadata/leader election; modern Kafka uses KRaft, a self-managed Raft quorum, removing the external dependency." Shows you're current.

Segments & retention

A partition isn't one giant file — it's split into segments (e.g. 1GB each). Only the active segment is written; older ones are immutable and indexed for fast offset lookup. Data is removed by policy, not by consumption:

  • Time/size retention — delete segments older than retention.ms (e.g. 7 days) or beyond retention.bytes.
  • Log compaction — keep only the latest value per key. Perfect for changelogs / "current state" topics (e.g. latest balance per account). Used by __consumer_offsets and Kafka Streams state.

Producers

The producer's job: pick a partition, batch records, and choose how strong a durability promise to wait for. Three knobs decide almost everything: acks, enable.idempotence, and batching.

acks — the durability ↔ latency dial

Play with it. Each setting trades safety against speed:

Durability
Latency
Throughput
Classic trap

acks=all alone does not guarantee durability. It means "all replicas in the ISR have it." If the ISR has shrunk to just the leader, "all" = 1. You must also set min.insync.replicas=2 so a write fails rather than silently landing on a single node.

Idempotent producer — no duplicates on retry

Networks fail. A producer sends a record, the broker writes it, the ack is lost, the producer retries → duplicate. Setting enable.idempotence=true gives each producer a PID and attaches a monotonic sequence number per partition. The broker rejects a record whose sequence it has already seen. Result: exactly-once delivery to a partition, even with retries. It's basically free and on by default in modern clients — always keep it on.

Transactions — atomic multi-partition writes

For "consume → process → produce" pipelines you need atomicity across multiple partitions/topics plus the offset commit. A transactional.id lets the producer wrap writes in beginTransaction() / commitTransaction(). Consumers with isolation.level=read_committed only see committed records. This is the basis of exactly-once stream processing in Kafka Streams.

Batching & compression — the throughput levers

props.put("acks", "all");
props.put("enable.idempotence", true);
props.put("linger.ms", 10);        // wait up to 10ms to fill a batch
props.put("batch.size", 32768);     // 32KB batches per partition
props.put("compression.type", "lz4"); // compress the batch
props.put("max.in.flight.requests.per.connection", 5);
  • linger.ms — a tiny delay to batch more records together. Higher = better throughput, slightly higher latency.
  • Compression applies to whole batches; bigger batches compress better. lz4/zstd are common.
  • max.in.flight > 1 without idempotence can reorder records on retry. With idempotence on, ordering is preserved up to 5 in-flight.
04 / FOUNDATIONS

Consumers & consumer groups

Consumers pull (not pushed to), in groups, and own their offsets. The whole model is built around the consumer controlling its own position in the log.

Groups & partition assignment

A consumer group is identified by group.id. Kafka assigns each partition to exactly one consumer within the group. Across different groups, everyone gets everything (fan-out).

  • 6 partitions, 3 consumers in group → 2 partitions each.
  • 6 partitions, 6 consumers → 1 each (max parallelism).
  • 6 partitions, 8 consumers → 6 work, 2 idle. Partitions cap your scale.
  • Two separate groups (analytics and billing) both read all 6 partitions independently.

Rebalancing — the thing that bites you

When a consumer joins/leaves/crashes, the group rebalances — partitions are reassigned. The classic "eager" protocol is stop-the-world: everyone revokes all partitions, then reassigns → a processing pause. Cooperative (incremental) rebalancing only moves the partitions that actually need to move, avoiding the global stall. Mention this — it's a real operational pain point.

Common cause of mystery rebalances

A consumer that takes too long between poll() calls (slow processing) misses max.poll.interval.ms, gets kicked out as "dead," triggering a rebalance. Fix: process faster, reduce max.poll.records, or raise the interval.

Offset management & delivery semantics

Offsets are stored in the internal __consumer_offsets topic. When you commit determines your delivery guarantee:

commit before process

At-most-once

Crash after commit, before processing → message lost. Rarely what you want.

commit after process

At-least-once

Crash after processing, before commit → reprocessed. The default & most common. Make consumers idempotent.

txn / dedup

Exactly-once

Via Kafka transactions (consume-process-produce) or an idempotent sink keyed on a dedup id.

The practical truth

Most real systems run at-least-once + idempotent consumers. You assume you might see a record twice and design the processing to be safe under replay (e.g. UPSERT by a unique event id). True end-to-end exactly-once is expensive — reach for it only when you must.

Consumer lag

Lag = log-end offset − consumer's committed offset. It's the health metric. Growing lag means consumers can't keep up with producers. You'll be asked how to fix it (answer in §06 / §11): add consumers up to the partition count, add partitions, speed up processing, or parallelise within a consumer.

05 / GOING DEEP

Why Kafka is so fast

A favourite interview question. Kafka pushes millions of messages/sec on commodity disks because of four design choices that all reinforce each other.

1. Sequential disk I/O

Append-only writes mean the disk head never seeks — sequential writes on spinning disk can rival random memory access. The log structure isn't an accident; it's the performance.

2. OS page cache

Kafka writes to the filesystem and lets the OS page cache hold hot data. Recent records are served from RAM without Kafka managing its own cache or fighting GC.

3. Zero-copy (sendfile)

To serve a consumer, data goes disk → NIC via sendfile() without copying through the application/user space. Huge CPU saving on reads.

4. Batching + compression

Producers batch records, compress the batch, and the broker stores it compressed and ships it compressed to consumers — amortising per-message overhead end to end.

One-liner

"Kafka is fast because it embraces the disk instead of fighting it: sequential appends, OS page cache, zero-copy reads, and batched/compressed transfers — so the same bytes are written, cached, and shipped with almost no extra copying."

06 / GOING DEEP

Scaling Kafka

Scaling is almost entirely a partition story. Partitions are the unit of parallelism, so capacity planning starts there.

Throughput math (back-of-envelope)

If a single consumer instance handles 10 MB/s and you need 100 MB/s, you need ≥ 10 consumers → ≥ 10 partitions. Always size partitions to your peak required consumer parallelism, with headroom for growth — because you can add partitions but you can't reduce them, and adding them changes key→partition mapping (breaks ordering for existing keys).

The "more partitions" tradeoff

More partitions → good

More consumer parallelism, more producer throughput, better load spread across brokers.

More partitions → costs

More open file handles & memory, longer leader-election/recovery on failure, more end-to-end latency, longer rebalances. Thousands per broker is a real limit (KRaft raises it).

Scaling the moving parts

  • Producers scale freely — just add more; they're stateless.
  • Consumers scale up to the partition count, then extras idle. To go further, add partitions or shard work downstream.
  • Brokers — add nodes and reassign partitions (kafka-reassign-partitions) to rebalance load. Plan for data movement.
  • Hot partition — a skewed key (one whale account = 50% of traffic) overloads one partition. Fixes: composite/salted key (accountId#bucket), or a custom partitioner. Trade some ordering for balance.
Fintech-flavoured example

Payments keyed by merchantId guarantee per-merchant ordering. But one mega-merchant on Black Friday creates a hot partition. Salt the key to merchantId#(orderId%4) to spread load — accepting that strict per-merchant ordering relaxes to per-(merchant,bucket).

07 / GOING DEEP

Failover & durability

Kafka's fault tolerance is replication + ISR + leader election. The durability you actually get depends on a small set of settings that must agree with each other.

What happens when a broker dies

  1. Followers stop receiving fetches from the dead leader.
  2. The controller detects the failure and elects a new leader for each affected partition from the ISR.
  3. Clients refresh metadata and reconnect to the new leaders. Brief unavailability, no data loss (if configured correctly).
  4. When the broker returns, its replicas re-sync from the current leaders and rejoin the ISR.

The durability contract (memorise this combo)

# Producer
acks = all
enable.idempotence = true
# Topic / broker
replication.factor = 3
min.insync.replicas = 2

This says: "A write is only acknowledged once at least 2 replicas have it. We can lose 1 broker with zero data loss and keep accepting writes. Lose 2 and we stop accepting writes (CP choice) rather than risk loss."

Unclean leader election — the availability vs durability fork

If all ISR replicas are down, do you (a) wait for an ISR member to recover — stay consistent, lose availability — or (b) elect an out-of-sync replica as leader and keep serving, losing the records it never received? That's unclean.leader.election.enable. Default false (durability first). For fintech, keep it false.

CAP positioning

Per partition, Kafka is effectively CP with the safe settings above: on a partition/failure it sacrifices availability (refuses writes when ISR < min) to avoid losing acknowledged data. Flip unclean.leader.election on and you've traded toward AP. Rack awareness (broker.rack) spreads replicas across racks/AZs so one zone failure doesn't take out all replicas of a partition.

08 / GOING DEEP

The tradeoffs, named

Senior answers sound like "it depends, here's the axis." These are the axes interviewers want you to name explicitly.

Durability ↔ Latency
acks=all + min.insync.replicas waits for replication → safer, slower. acks=1/0 is faster, riskier.
Ordering ↔ Parallelism
Ordering only holds within a partition. More partitions = more parallelism but no global order. Key choice decides which records stay ordered.
Throughput ↔ Latency
Bigger batches & higher linger.ms boost throughput but add per-message latency.
Availability ↔ Consistency
unclean.leader.election: serve stale-but-available, or refuse-but-consistent.
At-least-once ↔ Exactly-once
Exactly-once (transactions) costs throughput & complexity. At-least-once + idempotent consumer is cheaper and usually enough.
Partition count
More = parallel + balanced, but slower failover, more files, longer rebalances. Can't shrink.
Retention
Longer retention / replay power vs storage cost. Compaction keeps latest-per-key cheaply for state topics.
09 / APPLIED

Where Kafka is actually used

Pattern-match the use case to the design property it relies on — that's how you justify "why Kafka here."

Async microservice decoupling

Order service emits order.created; inventory, billing, notifications consume independently. Relies on fan-out + decoupling.

Event sourcing & CDC

The log is the system of record; rebuild state by replaying. Debezium streams DB row changes into Kafka. Relies on durable replay + compaction.

Stream processing

Kafka Streams / Flink do windowed aggregations, joins, fraud scoring directly on the streams. Relies on ordered partitions + exactly-once.

Log & metrics aggregation

Firehose of app logs/metrics into Kafka, fanned to Elasticsearch, S3, dashboards. Relies on raw throughput.

Activity / clickstream tracking

The original LinkedIn use case — every user action as an event for analytics & ML features.

Real-time fraud / risk

Stream of transactions/events scored against rules in real time. Replay lets you re-evaluate against new rules over historical data.

Your own story

A policy-based breach/fraud detection engine built on Kafka stream processing is a textbook "real-time risk on an ordered, replayable event stream" use case — exactly the kind of system this whole doc describes. Frame it that way and the design questions answer themselves.

The conversations that come up

Kafka vs RabbitMQ

Rabbit = smart broker / dumb consumer with flexible routing, per-message ack, and the message disappears when consumed. Kafka = dumb broker / smart consumer, retains everything, replayable, far higher throughput. Need routing & task queues → Rabbit. Need a durable replayable stream → Kafka.

Kafka vs Pulsar

Pulsar separates serving (brokers) from storage (BookKeeper bookies), giving independent scaling and easy tiered storage; native multi-tenancy and both queue + stream semantics. Kafka is more mature, larger ecosystem, simpler mental model.

Kafka vs Kinesis

Same log idea; Kinesis is fully managed AWS (shards instead of partitions, no ops). Pick Kinesis to avoid running clusters in an AWS shop; Kafka for portability, ecosystem, and higher ceiling.

Kafka vs SQS/SNS

SQS/SNS are dead-simple, infinitely elastic, zero-ops — but no replay, weak ordering, no stream processing. Great for plain decoupling; not for event sourcing or analytics streams.
