IN-MEMORY DATA STRUCTURE STORE

Redis, from zero to every-design-uses-it intuition.

Not "a cache." Redis is an in-memory, single-threaded data-structure server — strings, hashes, sorted sets, streams, all living in RAM and mutated by atomic commands. That's why it does millions of ops/sec, and why it shows up in nearly every system design: cache, rate limiter, leaderboard, lock, queue, presence.

01 / FOUNDATIONS

What Redis actually is

REmote DIctionary Server. A key-value store where the values are rich data structures, the whole dataset lives in memory, and a single thread executes commands one at a time.

The one-sentence model

Redis is a dictionary that lives in RAM, where each value can be a string, hash, list, set, sorted set, or stream — and every operation on it is atomic because one thread runs them in order.

Why it's blazingly fast

  • In-memory — no disk seek on the hot path; RAM access is orders of magnitude faster than disk.
  • Single-threaded command execution — no lock contention, no context switches, no race conditions between commands.
  • Efficient data structures — purpose-built encodings (ziplist, intset, skiplist) keep ops O(1)/O(log n).
  • I/O multiplexing (epoll) — one thread handles thousands of client connections via an event loop. Redis 6+ adds I/O threads for network reads/writes, but command execution stays single-threaded.

Core vocabulary

Key-value
Every entry is a key → a value, where the value is one of several data structures.
Atomic command
Each command runs to completion before the next starts — no interleaving.
TTL / expiry
Keys can be set to auto-expire after a time. Core to caching.
RDB / AOF
The two persistence options — snapshot vs append-only log.
Replica
A read-only copy that asynchronously follows a master.
Sentinel
The HA system that monitors and auto-fails-over masters.
Cluster
Sharding across nodes using 16384 hash slots.
Eviction
What Redis does when it hits its memory limit (LRU/LFU/etc.).

03 / FOUNDATIONS

The single-threaded model

Redis runs your commands on one thread. This is a feature, not a limitation — but it has a sharp edge you must respect.

Why single-threaded is good

  • Atomicity for free — no two commands interleave, so INCR, LPUSH, etc. are inherently thread-safe. No locks needed.
  • Predictable performance — no lock contention or thread scheduling jitter.
  • Simple consistency — the order commands arrive is the order they execute.
The sharp edge — one slow command blocks everything

Because it's one thread, a single O(N) command on a huge key freezes the whole server. KEYS * on millions of keys, FLUSHALL, big SMEMBERS/HGETALL, or a heavy Lua script will stall every other client. Use SCAN (cursor-based) instead of KEYS, avoid giant collections, and keep commands small.

Atomicity tools beyond single commands

  • Pipelining — send many commands without waiting for each reply, cutting network round-trips. Not atomic, just batched.
  • MULTI/EXEC (transactions) — queue commands and run them as one atomic block (no other client interleaves). No rollback though — it's atomic execution, not ACID rollback.
  • Lua scripts (EVAL) — run custom logic server-side atomically. Used for correct rate limiters and compare-and-set patterns. Keep them fast (they block the thread).
# Atomic "check-and-increment" rate limiter via Lua — runs as one unit
EVAL "local c = redis.call('INCR', KEYS[1])
       if c == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end

04 / GOING DEEP

Persistence: RDB vs AOF

Redis is in-memory, but it can persist to disk so a restart doesn't lose everything. Two mechanisms, different durability/performance tradeoffs — a classic interview question.

snapshot

RDB (snapshotting)

Periodically fork and dump the whole dataset to a compact binary file. Fast restarts, small files, great for backups. But you lose everything written since the last snapshot if Redis crashes.

log

AOF (append-only file)

Logs every write command to a file. Replayed on restart to rebuild state. Much more durable (configurable fsync: every write / every second / OS-decides), but bigger files and slower restart.

How to reason about it

  • Pure cache (data is rebuildable) → persistence optional; even RDB off is fine. Speed first.
  • Can tolerate seconds of loss → AOF with everysec fsync (the common balanced choice) or RDB.
  • Near-zero loss needed → AOF with always fsync — slowest, most durable. Often you also run replicas.
  • Best of both → run RDB + AOF together; AOF for durability, RDB for fast restart/backups.
The honest caveat

Even AOF everysec can lose up to ~1 second of writes on crash, and replication is asynchronous (§06). So Redis is not a system of record for money. Treat it as a fast layer in front of a durable database, not the durable database.

05 / GOING DEEP

Eviction & expiry

RAM is finite. When Redis hits maxmemory, the eviction policy decides what to drop. Picking the right one is a real interview detail.

How keys expire

  • Lazy expiry — an expired key is removed when next accessed.
  • Active expiry — a background job samples and removes expired keys periodically, so dead keys don't linger forever unused.

maxmemory eviction policies

noeviction
Reject writes when full (return errors). Safe for a store you can't lose data from; bad for a cache.
allkeys-lru
Evict least-recently-used across all keys. The default go-to for a general cache.
allkeys-lfu
Evict least-frequently-used. Better when popularity matters more than recency.
volatile-lru/lfu/ttl
Only evict keys that have a TTL set — keeps non-expiring keys safe. volatile-ttl drops those nearest expiry first.
allkeys-random
Evict random keys. Cheap, rarely ideal.
Quick rule

Pure cache where anything can be re-fetched → allkeys-lru (or allkeys-lfu for skewed popularity). Mixing cache + must-keep data in one instance → use volatile-* and set TTLs only on the cache keys. Using Redis as a data store → noeviction and size memory carefully.

06 / GOING DEEP

Replication, HA & Cluster

Three separate concepts people conflate: replication (copies), Sentinel (automatic failover), and Cluster (sharding for scale). Keep them distinct in your answer.

Replication — read scaling + redundancy

A master accepts writes; one or more replicas asynchronously copy its data and serve reads. This scales reads and gives you a standby. Asynchronous is the key word: a master can ack a write and crash before the replica receives it → that write is lost on failover.

Sentinel — automatic failover

Replication alone doesn't promote a replica when the master dies — something must detect failure and reconfigure. Redis Sentinel is a set of processes that monitor the master, reach quorum that it's down, elect and promote a replica to master, and tell clients the new address. It gives high availability for a single-shard (non-sharded) setup.

Cluster — horizontal sharding

When one node can't hold the dataset or handle the write throughput, Redis Cluster shards data across nodes using 16384 hash slots. slot = CRC16(key) % 16384; each node owns a range of slots. Each shard has its own master + replicas, so Cluster gives scaling and HA together.

# key → slot → node
CRC16("user:42") % 16384  →  slot 866  →  node B (owns 0–5460)
The multi-key gotcha in Cluster

Multi-key commands (e.g. MGET, transactions, SUNION) only work if all keys live in the same slot. Use hash tags — put a {...} in the key so only that part is hashed: {user:42}:profile and {user:42}:cart land in the same slot. Without this, cross-slot multi-key ops fail.

Say this clearly

"Replication = copies for reads/redundancy. Sentinel = automatic failover for a single shard. Cluster = sharding across many shards, each itself replicated." Distinguishing the three cleanly signals real understanding.

07 / GOING DEEP

The tradeoffs, named

Name the axis — that's the senior move.

Speed ↔ Durability
In-memory + async persistence/replication is fast but can lose recent writes. Not a system of record for critical data.
Memory ↔ Cost
Everything in RAM means dataset size is bounded by (expensive) memory. Big datasets cost real money or force eviction.
Single-thread ↔ Throughput
No locks/atomic, but one slow command blocks all; CPU-bound work doesn't parallelise within a shard. Scale via Cluster sharding.
Consistency ↔ Availability
Async replication + failover can drop acked writes (replicas may lag). It favours availability/speed over strict consistency.
Cache freshness ↔ Hit rate
Short TTLs = fresher but more misses/DB load; long TTLs = higher hit rate but staler data.
Simplicity ↔ Cluster complexity
Single instance is dead simple; Cluster adds slot management, hash-tag constraints, and client awareness.

The three cache failure modes (know all three)

Penetration

Queries for keys that don't exist always miss cache and hit the DB. Fix: cache the "not found" result (short TTL), or a Bloom filter to reject unknown keys.

Avalanche

Many keys expire at the same moment (or Redis restarts) → a flood of misses stampedes the DB. Fix: jitter the TTLs, and protect the DB with limited concurrency.

Stampede / hot key

One very popular key expires; thousands of requests miss and recompute simultaneously. Fix: a mutex/lock so one request recomputes while others wait or serve stale; or refresh-ahead before expiry.

09 / APPLIED

Distributed primitives

Beyond caching, Redis is the Swiss-army knife for distributed-systems building blocks. The ones interviewers expect:

Rate limiter

Fixed-window: INCR a per-user key with an EXPIRE — if it exceeds the limit, reject. Sliding-window/token-bucket: a small Lua script for atomic correctness. Cheap, distributed, and exactly the kind of thing a fraud/abuse system needs.

Distributed lock

A basic lock: SET key uniqueToken NX PX 30000 — set only if absent, with an auto-expiry so a crashed holder doesn't deadlock. Release by checking the token and deleting (atomically, via Lua). For multi-node correctness there's Redlock, though it's debated — for most apps a single-instance lock with fencing tokens is enough.

Lock honesty

Redis locks are best-effort, not a consensus system. Under failover/clock issues two clients can briefly both hold a "lock." For correctness-critical mutual exclusion, add fencing tokens or use a real consensus store (ZooKeeper/etcd). Say this — it shows maturity.

Pub/Sub & Streams

  • Pub/Sub — fire-and-forget messaging. No persistence; if a subscriber is offline, it misses the message. Great for a WebSocket backplane / live fan-out where loss is acceptable.
  • Streams — an append-only log with consumer groups (Kafka-lite). Persistent, replayable, supports acks. Use when you need a durable lightweight queue without standing up Kafka.

Redis vs Memcached

Memcached is multi-threaded and dead-simple for plain string caching. Redis wins on data structures, persistence, replication, pub/sub, scripting, and Cluster. Pick Memcached only for the narrowest "fast string cache, nothing else"; Redis for almost everything else.

Redis vs in-process cache

Local cache (Caffeine/Guava) has zero network latency but lives per-node — each server has its own copy, hard to invalidate consistently. Redis is a shared, consistent cache across all nodes. Often you layer both: local L1 + Redis L2.

Redis Streams vs Kafka

Streams = lightweight durable queue when Redis is already in your stack and volumes are modest. Kafka = purpose-built for very high throughput, long retention, and big consumer fan-out. Don't replace Kafka with Streams at scale.

Redis as primary DB?

Possible with AOF + replication, but async durability means it's risky for money/orders. Use it as a fast layer in front of a durable store, not the source of truth — say this explicitly in fintech rounds.
