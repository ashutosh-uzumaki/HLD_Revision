REAL-TIME, FULL-DUPLEX TRANSPORT

WebSockets, from zero to scaled-to-millions intuition.

HTTP is a one-shot question-and-answer. A WebSocket is a single, persistent, full-duplex connection — once open, client and server push messages to each other any time, with almost no per-message overhead. The hard part isn't opening one; it's running a million of them. That's the interview.

01 / FOUNDATIONS

What WebSockets actually are

A WebSocket is a bidirectional, persistent communication channel over a single TCP connection, established by upgrading an HTTP request. After the upgrade, the HTTP semantics are gone — it's a lightweight framed message protocol.

The one-sentence model

Start with a normal HTTP request, ask the server to "upgrade" the connection, and if it agrees, the same TCP socket stays open as a two-way pipe — either side can send a message at any moment without a new request.

The problem it solves

Plain HTTP is client-initiated and request-response: the server can't speak unless asked. For real-time features (chat, live prices, notifications) you'd otherwise hammer the server with repeated requests (polling), wasting bandwidth and adding latency. WebSockets flip this: one connection, server can push instantly.

  • Full-duplex — both directions simultaneously, unlike request-response.
  • Persistent — opened once, reused for the session (no repeated handshakes/headers).
  • Low overhead — a few bytes of framing per message vs full HTTP headers each time.
  • Low latency — no connection setup per message; the pipe is already open.

Core vocabulary

ws:// / wss://
The WebSocket URI schemes. wss = WebSocket over TLS (always use it in prod).
Upgrade handshake
The HTTP request that switches the connection from HTTP to WebSocket.
Frame
The unit of data after the handshake — text, binary, ping, pong, or close.
Full-duplex
Both sides can send at the same time, independently.
Heartbeat
Periodic ping/pong frames to keep the connection alive and detect dead peers.
Backplane
A pub/sub layer (Redis/Kafka) letting many WS servers broadcast to each other's clients.
Sticky session
Load-balancer setting pinning a client to the same server for the connection's life.
First thing to say in an interview

"A WebSocket connection is stateful and long-lived — that single property is the source of every scaling and reliability challenge that follows." It frames the whole conversation.

02 / FOUNDATIONS

The handshake

A WebSocket begins life as an ordinary HTTP/1.1 GET with special headers. The server agrees with a 101 Switching Protocols, and from that instant the bytes on the wire are WebSocket frames, not HTTP.

Client → Server (the upgrade request)

GET /socket HTTP/1.1
Host: chat.app
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==   # random base64 nonce
Sec-WebSocket-Version: 13
Sec-WebSocket-Protocol: chat.v1            # optional subprotocol

Server → Client (acceptance)

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=  # = base64(sha1(key + magic GUID))

The Sec-WebSocket-Accept value is derived from the client's key plus a fixed "magic string." This isn't security — it just proves both sides actually speak the WebSocket protocol and a confused HTTP cache/proxy didn't blunder in. After the 101, the TCP socket is repurposed for framed messages.

Why piggyback on HTTP at all?

Because port 80/443 and HTTP infrastructure (proxies, load balancers, firewalls) already exist everywhere. Upgrading from HTTP means WebSockets traverse the same paths as normal web traffic — especially over wss/443, which proxies tend to pass through untouched.

03 / FOUNDATIONS

Frames & heartbeats

After the handshake, data moves in small frames. Understanding frame types and heartbeats is what separates "I used the browser API" from "I understand the protocol."

Frame types (opcodes)

data

Text / Binary

The actual payload. Text = UTF-8, Binary = raw bytes (e.g. protobuf, media).

control

Ping / Pong

Heartbeats. A ping must be answered with a pong. Used to keep-alive and detect dead peers.

control

Close

Graceful shutdown frame with a status code; both sides exchange it to close cleanly.

Two protocol details that come up

  • Masking — every client→server frame is XOR-masked with a random key. It's not encryption; it prevents certain cache-poisoning attacks against intermediaries. Server→client frames are not masked.
  • Fragmentation — a large message can be split across multiple frames and reassembled, so you don't have to buffer a giant payload before sending.

Heartbeats — the part everyone forgets

A WebSocket can silently die: a laptop sleeps, Wi-Fi drops, a NAT/load-balancer times out an idle connection. TCP won't always tell you quickly. So you send periodic ping frames and expect pong back within a timeout; no pong → assume dead, close, and (on the client) reconnect.

Real production bug

Load balancers and proxies kill idle connections (often 60s). If your app is quiet, the LB silently drops the socket. Heartbeats every ~30s keep it "active" and surface death fast. Forgetting this = mysterious disconnects in prod.

04 / THE BIG COMPARISON

The real-time protocol family

WebSockets are one of five ways to do "real-time" on the web. Interviewers love asking you to pick the right one — and the right answer is often not WebSockets. Toggle through them:

Real-time freshness
Server efficiency
Bidirectional
Simplicity
The decision rule

Server-push only (feeds, notifications, live dashboards)? → SSE — simpler, auto-reconnects, runs over plain HTTP. True two-way, low-latency (chat, games, collab editing)? → WebSocket. Peer-to-peer media/low-latency between users (video calls)? → WebRTC. Infrequent, simple, don't-care-about-instant? → polling. Don't reach for WebSockets by default.

The conversations that come up

WebSocket vs SSE

SSE is one-way (server→client), text-only, over plain HTTP, with built-in auto-reconnect and event IDs. WebSocket is two-way and binary-capable but you implement reconnection yourself. If the client rarely needs to send (and can use a normal POST when it does), SSE is simpler and often the better choice.

WebSocket vs Long polling

Long polling fakes push by holding a request open until there's data, then the client immediately re-requests. Works everywhere, but every message costs a full HTTP round-trip and re-connect. WebSockets keep one pipe open — far less overhead at high message rates.

WebSocket vs WebRTC

WebRTC is peer-to-peer over UDP for the lowest latency media — but needs signaling (often via WebSockets!), STUN/TURN for NAT traversal, and is complex. Use WebRTC for video/voice between users; WebSocket for client↔server messaging.

Why not HTTP/2 push?

Server Push was for pre-pushing page assets, not app messaging, and is effectively deprecated. It's not a real-time messaging transport — don't propose it.

06 / GOING DEEP

Scaling to millions of connections

This is the heart of any WebSocket system-design round. The core difficulty: connections are stateful and long-lived, so you can't just round-robin requests across stateless servers like normal HTTP.

The fan-out problem

User A is connected to server-1. User B is connected to server-2. A sends a message for B. Server-1 holds no socket to B — so how does it reach B?

The answer: a pub/sub backplane

Servers don't talk to clients they don't hold. Instead, server-1 publishes the message to a shared pub/sub layer (Redis Pub/Sub, or Kafka for durability). Every WS server subscribes; server-2 receives it and pushes it down its socket to B. The backplane decouples "which server holds whom."

A (on server-1) → "hi B"
server-1  ── publish topic:userB ──▶  Redis / Kafka  ──▶  server-2  ──▶  socket(B)
                                          │
                                          └── all WS servers subscribe

The pieces of a scalable WS layer

  • Connection (gateway) tier — stateless-ish servers whose only job is holding sockets and (un)marshalling frames. Keep business logic out of here.
  • Pub/sub backplane — Redis Pub/Sub (fast, fire-and-forget) or Kafka (durable, replayable) to route messages between servers.
  • Connection registry / presence — a store (often Redis) mapping userId → which server holds them, for targeted routing and online/offline status.
  • Load balancer — L4 (TCP) preferred for WS; supports the upgrade and long-lived connections. Use sticky sessions so a reconnect/upgrade stays on one server.

The per-server limits (C10K → C10M)

One process can hold far more idle WebSockets than you'd think — but you hit real ceilings:

  • File descriptors — each connection is an open FD; tune ulimit / OS limits. The classic blocker.
  • Memory per connection — buffers + app state. With efficient event-loop servers (Node, Netty, Go, epoll-based) you can hold 100k–1M idle connections per box; the limit is memory and FDs, not CPU when idle.
  • CPU on message throughput — idle is cheap; serializing/broadcasting to many is what burns CPU. Fan-out amplification (one message → 100k sends) is the real cost.
  • Horizontal scale — add gateway servers behind the LB; the backplane lets them cooperate. This is how you go from 100k to 10M.
Broadcast amplification trap

A "stock price" sent to 1M subscribers = 1M individual frame writes per tick. At high tick rates this melts servers. Mitigations: coalesce/batch updates (send every 200ms not every tick), shard subscribers across servers, and only send deltas.

07 / GOING DEEP

Reliability & reconnection

Connections will drop — networks flake, servers deploy, LBs time out. A production WebSocket system is defined by how gracefully it handles disconnects without losing messages.

Detecting death

Heartbeats (ping/pong) on a timer. No pong within the window → declare dead, clean up server-side state (free the FD, update presence), and the client triggers reconnect.

Reconnection — done right

  • Exponential backoff + jitter — don't retry instantly in a tight loop. Back off (1s, 2s, 4s…) with randomness so 1M clients don't reconnect in lockstep.
  • Resume, don't restart — on reconnect, re-authenticate and re-subscribe to the rooms/channels the user was in.
The thundering-herd reconnect storm

Deploy a server holding 200k connections → all 200k reconnect at once → they stampede the remaining servers and the auth service, causing cascading failure. Fixes: jittered backoff, gradual connection draining on deploy, and rate-limiting reconnect/auth.

Not losing messages across a drop

WebSocket frames over TCP are in-order within a connection. But if the connection drops, messages sent during the gap are lost unless you plan for it:

  • Sequence numbers / message IDs — the client tracks the last id it saw; on reconnect it sends "I have up to id N."
  • Catch-up — the server replays missed messages from a durable store (DB, or a Kafka log keyed by room) since id N, then resumes live delivery. This is exactly where a durable log behind the backplane earns its keep.
  • Acks — for critical messages, the client acks receipt; unacked messages are re-sent.

Graceful deploys

On shutdown, send a close frame, let the LB stop routing new connections (drain), and rely on client backoff to spread reconnects. Never just kill the process and dump every socket simultaneously.

08 / GOING DEEP

Auth & security

WebSockets have sharp edges around auth because the browser API is limited and the connection is long-lived.

Authenticating the handshake

The browser limitation everyone trips on

The browser WebSocket API cannot set custom headers (no Authorization: Bearer). So you authenticate by: (1) a short-lived token in the query string (wss://app/socket?token=...), (2) the Sec-WebSocket-Protocol subprotocol header as a smuggle channel, or (3) an existing session cookie sent automatically on the handshake. Validate during the upgrade and reject before completing it.

The other essentials

  • Always wss (TLS) — encrypts frames and helps traverse proxies cleanly.
  • Origin checking — verify the Origin header on the handshake to block cross-site WebSocket hijacking (CSWSH); CORS does not protect WebSockets.
  • Authorization per message — auth at connect isn't enough; check the user is allowed to send to this room/channel on each message.
  • Rate limiting & quotas — cap messages/sec and connections per user; a long-lived socket is a cheap DoS vector otherwise.
  • Token expiry on long connections — a socket can outlive its token. Re-validate periodically or force reconnect on expiry.
09 / GOING DEEP

The tradeoffs, named

Name the axis. That's what makes an answer sound senior.

Stateful ↔ Scalable
Persistent connections give low latency but pin state to servers — needing sticky sessions, a backplane, and presence tracking. Stateless HTTP scales more trivially.
Push freshness ↔ Server cost
Holding millions of open sockets costs FDs/memory even when idle. Polling is stateless but wastes requests. Pick per traffic pattern.
Bidirectional ↔ Simplicity
WebSocket gives two-way but you own reconnection, heartbeats, auth quirks. SSE/polling are simpler when you don't need true duplex.
Latency ↔ Reliability
Fire-and-forget (Redis pub/sub) is fast but can drop on the gap; durable replay (Kafka) guarantees catch-up but adds latency/complexity.
Broadcast reach ↔ CPU
Fan-out to many subscribers amplifies one message into N writes. Batch/coalesce to protect servers.
Connection limits ↔ Cost
More connections per box = cheaper, but raises blast radius when that box dies (more reconnects at once).
When NOT to use WebSockets

One-way updates → SSE. Rare/occasional updates → polling. Request-response that happens to be frequent → just HTTP/2. Reaching for WebSockets when you don't need duplex adds statefulness and ops burden for no benefit. Saying this unprompted scores points.

10 / APPLIED

Use cases & reference architecture

Match the use case to "why duplex / why persistent" — that's the justification interviewers want.

Chat & messaging

WhatsApp/Slack-style. Two-way, presence, typing indicators, delivery acks. The canonical WebSocket system-design problem.

Live trading / price tickers

Server pushes price updates at high frequency; client can place orders on the same pipe. Low latency is the whole point (Zerodha-style).

Collaborative editing

Google-Docs-style. Rapid bidirectional ops (cursor, edits) needing low latency and ordering per document.

Live dashboards & notifications

Often SSE is enough (server→client only). Use WS only if the client must also send frequently.

Multiplayer games

Fast state sync. Sometimes WebSocket for lobby/state, WebRTC/UDP for the latency-critical game loop.

Live sports / auctions

Real-time scores, bids. Fan-out heavy — batch updates to avoid broadcast amplification.

Reference architecture for a chat backbone

clients ──wss──▶ L4 LB (sticky) ──▶ [ WS gateway servers (hold sockets) ]
                                              │            ▲
                              publish/subscribe│            │push
                                              ▼            │
                                   Redis Pub/Sub  /  Kafka  ◀── message/persistence service
                                              │
        presence (userId→server) ◀── Redis    └── durable msg log (catch-up on reconnect)

Gateway tier holds connections and is dumb. A message service handles persistence, ordering, and history. The backplane routes between gateways. Presence in Redis answers "is user online / which server." On reconnect, the client catches up from the durable log via last-seen message id.

Tie-in to your stack

This is the same shape as the Kafka doc's "real-time event stream" — the WS gateway is the edge, and a durable log (Kafka) behind the backplane is what makes reconnection lossless. The two technologies compose: WebSockets at the edge, Kafka for durability/fan-out.
